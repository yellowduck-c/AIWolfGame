# 前后端接口说明

本文档基于当前代码实际状态，整理 `werewolf_frontend` 与 `werewolf_backend` 之间的接口与协议约定。

本文档明确区分 4 类内容：

- **已实现接口**：当前前后端代码里已经真实存在
- **前端已定义协议**：前端已经发送/消费，但后端未必完整保证
- **前端本地预留协议**：当前只在前端本地生效，还没有接入真实后端
- **待对齐项**：前后端后续必须保持一致的字段和行为

---

## 1. 总览

当前项目的前后端通信分为两类：

- **HTTP**：用于服务存活检查与基础探活
- **WebSocket**：用于观战页实时连接、控制命令发送与事件推送

当前代码实际情况如下：

### 1.1 当前后端已实现

- `GET /`
- `GET /api/health`
- `WS /ws`
- WebSocket 连接成功后推送 `CONNECTED`
- WebSocket 收到任意文本后回显 `ECHO`

### 1.2 当前前端已定义并使用

前端已经通过 WebSocket 发送以下控制命令：

- `GAME_START`
- `GAME_PAUSE`
- `GAME_STOP`

前端已经编写了以下事件处理逻辑：

- `GAME_STARTED`
- `AGENT_SPEAK`
- `AGENT_STATUS_CHANGE`
- `PHASE_CHANGE`
- `GAME_OVER`
- `CONNECTED`
- `ECHO`

### 1.3 当前仍需注意

- 前端发送的 `GAME_START / GAME_PAUSE / GAME_STOP` 已经是正式消息格式，但后端是否完整按同一协议处理，要以当前后端代码为准
- 人机混战面板里的“人类发言 / 投票 / 技能”目前主要是前端本地交互，不应误认为是已上线的后端协议


#### 说明

- 返回字段已经不止早期版本里的 `status/app/port` 3 个字段，文档应以当前代码为准

### 4.2 当前已实现的 WebSocket 传输行为

当前后端 WebSocket 已真实实现的行为如下。

| 状态 | 方向 | 名称 | 当前行为 | 说明 |
| --- | --- | --- | --- | --- |
| 已实现 | 后端 -> 前端 | `CONNECTED` | 建立连接后主动发送 | 表示 WebSocket 已就绪 |
| 已实现 | 前端 -> 后端 | 任意文本消息 | 后端读取原始文本 | 当前按文本接收，不强制解析 JSON |
| 已实现 | 后端 -> 前端 | `ECHO` | 把收到的原始文本包装后回传 | 当前主要用于链路验证 |

#### `CONNECTED` 返回格式

```json
{
  "event": "CONNECTED",
  "message": "WebSocket ready"
}
```

#### `ECHO` 返回格式

```json
{
  "event": "ECHO",
  "payload": "<原始文本>"
}
```

#### 说明

- 当前 `ECHO` 属于调试/占位行为，不代表完整对局协议
- 这部分属于“当前真实后端传输层行为”，应与后面的“前端游戏协议”分开看

## 5. 前端当前实际发送的 WebSocket 命令

文件：`werewolf_frontend/src/utils/websocket.ts`

前端当前真实发送以下 3 个控制命令。

| 状态 | 方向 | 命令 | 消息格式 | 前端发送位置 | 备注 |
| --- | --- | --- | --- | --- | --- |
| 前后端已对齐 | 前端 -> 后端 | `GAME_START` | `{ "cmd": "GAME_START", "player_count": number, "role_counts": { ... } }` | `sendGameCommand()` / `GameBattle.vue` | 按前端配置创建 AI 对局 |
| 前端已发送 | 前端 -> 后端 | `GAME_PAUSE` | `{ "cmd": "GAME_PAUSE", "game_id"?: string }` | `sendGameCommand()` / `GameBattle.vue` | 暂停对局 |
| 前端已发送 | 前端 -> 后端 | `GAME_STOP` | `{ "cmd": "GAME_STOP", "game_id"?: string }` | `sendGameCommand()` / `GameBattle.vue` | 结束对局 |

### 5.1 开始对局

```json
{
  "cmd": "GAME_START",
  "player_count": 10,
  "role_counts": {
    "werewolf": 3,
    "seer": 1,
    "witch": 1,
    "hunter": 1,
    "villager": 4
  }
}
```

#### 当前字段含义

- `player_count`: 本局玩家总人数
- `role_counts`: 前端当前配置的角色数量
  - `werewolf`: 狼人数量
  - `seer`: 预言家数量
  - `witch`: 女巫数量
  - `hunter`: 猎人数量
  - `villager`: 村民数量

#### 当前对齐要求

- 后端必须使用前端传入的 `player_count` 和 `role_counts` 创建本局 AI agent
- `role_counts` 各项总和必须等于 `player_count`
- 当前角色键名以英文固定值传输，后端再映射为中文角色与阵营

### 5.2 暂停对局

```json
{
  "cmd": "GAME_PAUSE",
  "game_id": "<当前对局ID>"
}
```

### 5.3 结束对局

```json
{
  "cmd": "GAME_STOP",
  "game_id": "<当前对局ID>"
}
```

### 5.4 当前对齐说明

- 前端已经固定按 JSON 对象发送控制命令
- 后端应把 `cmd` 视为当前控制协议的统一入口
- `GAME_START` 不再只是“空启动命令”，而是当前对局配置的正式入口
- 如果后端仍忽略 `player_count / role_counts`，则说明前后端尚未真正按配置联动

## 8. 当前必须对齐的字段与行为

这一节描述的是前后端必须保持一致的实际协议约束。

### 8.1 WebSocket 消息编码

当前前端发送方式：

```ts
socket.send(JSON.stringify(payload))
```

因此建议统一约定：

- 前端 -> 后端控制消息使用 JSON 对象
- 后端应解析 `cmd`
- 不再把“纯文本回显”当作正式业务协议

### 8.2 控制命令入口字段

当前前端统一使用：

- `cmd`

因此后端控制协议建议统一以 `cmd` 为入口，而不是多套不一致字段。

### 8.3 `GAME_STARTED` 的玩家字段

前端当前要求每个玩家对象至少包含：

- `id`
- `role`
- `camp`
- `status`

这 4 个字段应视为当前观战页面的最低兼容要求。

### 8.4 `GAME_START` 的配置字段

前端当前会在启动对局时传入：

- `player_count`
- `role_counts`
  - `werewolf`
  - `seer`
  - `witch`
  - `hunter`
  - `villager`

后端应使用这些字段创建本局实际 AI 玩家，而不是再忽略前端配置回退到固定默认阵容。

### 8.5 阶段枚举值

当前前端稳定识别的阶段值：

- `night`
- `day_speech`
- `voting`
- `finished`

后端应优先使用这些值。

### 8.6 玩家状态枚举值

当前前端稳定识别的状态值：

- `alive`
- `dead`
- `exiled`

后端应优先使用这些值。

### 8.7 `GAME_STARTED` 与 `PHASE_CHANGE` 的关系

当前前端在 `GAME_STARTED` 后会进入实时对局态，但真正阶段展示仍以后续 `PHASE_CHANGE` 为准更稳妥。

推荐约定：

- `GAME_STARTED` 负责初始化对局与玩家快照
- `PHASE_CHANGE` 负责驱动阶段变化

这样前后端职责更清晰。

### 8.8 未来人机混战的扩展字段

当前 `initializeAgents()` 会默认把所有玩家展示为 AI，并直接显示真实身份与阵营。

如果后续要正式支持人机混战，应补充至少一类信息：

- `playerType`
- `humanPlayerId`
- 或更完整的“可见身份 / 可见阵营”字段

否则前端无法正确区分“观战 god-view”与“人类玩家视角”。

## 10. 结论

当前项目已经有了一个可以持续演进的前后端接口骨架，但文档必须区分清楚“当前已实现”和“前端已预留/已依赖”。

现在可以明确归纳为：

- **HTTP 基础接口已实现**

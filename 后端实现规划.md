# 后端实现规划

本文档用于指导 `werewolf_backend` 的下一步实现，目标是在当前前端已经成型、协议已经初步固定的前提下，尽快把后端从“占位骨架”推进到“可驱动完整狼人杀流程”的正式实现。

本版规划在原有 FastAPI + WebSocket + SQLite 的基础上，进一步明确引入 **Redis 作为缓存层**。后端后续推荐采用：

- **Redis**：活跃对局运行态、会话控制、临时动作、等待输入状态
- **SQLite**：结构化日志、对局结果、历史记录、复盘数据

这意味着后端将不再仅依赖 SQLite，也不建议把运行中对局完全留在进程内内存中，而是采用更适合实时系统的 **Redis + SQLite 双层架构**。

---

## 1. 当前后端现状

目前后端仅完成了最小骨架：

- `werewolf_backend/main.py`
  - 注册 FastAPI 应用
  - 注册 HTTP 路由与 WebSocket 路由
- `werewolf_backend/api/routes.py`
  - 实现 `GET /api/health`
- `werewolf_backend/websocket/routes.py`
  - 实现 `WS /ws`
  - 当前仅支持：
    - 连接成功后发送 `CONNECTED`
    - 收到消息后回显 `ECHO`
- `werewolf_backend/config/settings.py`
  - 已支持基础应用配置
  - 已支持 `database_url`
  - 当前尚未加入 Redis 连接配置
- `werewolf_backend/requirements.txt`
  - 当前已包含 FastAPI / SQLAlchemy / WebSocket 基础依赖
  - 当前尚未加入 Redis 客户端依赖

也就是说，当前后端还**没有真正的游戏引擎、对局状态、角色逻辑、Agent 调度、缓存层、日志持久化**。

但好处是：前端协议骨架已经明确。

可直接参考：
- [前后端接口说明.md](前后端接口说明.md)

后端下一步应围绕这份接口说明，优先补齐协议驱动能力，并同步建立 Redis + SQLite 的后端基础设施。

---

## 2. 后端实现目标

后端最终需要承担以下职责：

1. 创建并维护对局状态
2. 执行完整狼人杀流程
3. 调度不同角色 Agent 在正确阶段行动
4. 严格隔离公共信息 / 阵营共享信息 / 私有角色信息
5. 向前端持续推送结构化事件
6. 使用 Redis 管理活跃对局运行态
7. 使用 SQLite 记录完整对局日志，支持复盘与评估
8. 在后续阶段支持人类玩家输入接入

换句话说，后端不是“前端的数据源”而已，而是整个项目的：

- 规则中枢
- 状态中枢
- 调度中枢
- 缓存中枢
- 日志中枢

## 3. 核心架构建议：Redis + SQLite 双层数据模型

## 3.1 Redis 的职责

Redis 用于保存**活跃对局的运行时状态**，也就是当前仍在进行中的游戏数据。

建议存储内容：

- `game_id -> active session` 映射
- 当前阶段
- 当前轮次
- 活跃玩家状态快照
- 投票临时结果
- 夜间技能待结算动作
- 暂停 / 停止标志
- 等待人类输入标志
- 调度相关短期状态

Redis 在这里扮演的是：
- 快速读写层
- 活跃会话缓存层
- 调度协调层

## 3.2 SQLite 的职责

SQLite 用于保存**可追溯的持久化数据**。

建议存储内容：

- 对局元数据
- 玩家与角色分配结果快照
- 结构化事件日志
- 阶段流转记录
- 最终胜负结果
- 复盘与查询所需历史记录

SQLite 在这里扮演的是：
- 历史记录层
- 结果归档层
- 复盘查询层

## 3.3 两者的关系

建议明确规定：

- **Redis 是活跃对局运行时真值源**
- **SQLite 是历史记录真值源**

即：
- 游戏进行中，实时状态优先从 Redis 读取
- 游戏结束后，关键事实必须已经落到 SQLite

## 3.4 明确不要放进 Redis 的内容

以下内容不应只存在于 Redis：

- 最终胜负结果
- 历史事件唯一记录
- 唯一的角色分配记录
- 唯一可复盘数据

另外也不要把以下对象直接放进 Redis：

- WebSocket 连接对象本身
- Python 进程内不可序列化对象

WebSocket 连接应仍由当前进程内的连接管理器持有，Redis 只保存与连接/会话相关的元数据。

## 4. 实现原则

### 4.1 协议优先

先保证后端能稳定支持当前前端已经使用的协议，再向内补游戏逻辑。

优先对齐：

- `GET /`
- `GET /api/health`
- `WS /ws`
- WebSocket 命令：
  - `GAME_START`
  - `GAME_PAUSE`
  - `GAME_STOP`
- WebSocket 事件：
  - `GAME_STARTED`
  - `AGENT_SPEAK`
  - `AGENT_STATUS_CHANGE`
  - `PHASE_CHANGE`
  - `GAME_OVER`

### 4.2 先做纯 AI 对战

按照课题要求与当前项目节奏，优先跑通：

- 纯 AI 对战
- 完整回合循环
- 前端观战展示

人机混战放在核心流程稳定之后接入。

### 4.3 Redis 先于复杂 Agent 落地

在真正引入复杂 Agent / LLM 行为之前，先把：

- Redis 会话缓存层
- 可运行游戏状态机
- WebSocket 事件广播

做稳定。

这样后续接 Agent 决策时，不会反过来重构底层状态系统。

### 4.4 日志不是附属品

日志必须从后端早期就纳入设计。

因为课题要求明确强调：
- 可观测性
- 结构化记录
- 对局复盘
- 评测与对比能力

### 4.5 Redis 是基础设施，不是后期优化

本项目既然已经明确需要缓存，就应在架构层直接承认：

- Redis 不是“以后再加”的增强项
- Redis 是活跃对局管理的基础设施

这样设计更利于后续：
- 多连接会话管理
- 暂停 / 恢复
- 等待人类输入
- 后期可能的多进程扩展

## 5. 推荐实现阶段

## 阶段一：协议稳定与基础链路打通

### 目标

让当前前端可以和后端形成一个真正可跑通的闭环。

### 本阶段要完成的内容

1. 保持现有接口路径不变：
   - `GET /`
   - `GET /api/health`
   - `WS /ws`

2. 在 `/ws` 中加入命令解析：
   - 接收 JSON
   - 识别 `cmd`
   - 根据命令分发处理逻辑

3. 支持以下最小命令：
   - `GAME_START`
   - `GAME_PAUSE`
   - `GAME_STOP`

### 本阶段结果

完成后应达到：
- 前端点击“开始”后，不再只是收到回显
- WebSocket 已经具备真正的命令分发骨架

---

## 阶段二：建立 Redis 缓存层与会话管理器

### 目标

在游戏引擎正式展开之前，先把活跃对局的缓存基础设施搭好。

### 推荐扩展位置

- `werewolf_backend/config/settings.py`
  - 增加 Redis 配置，例如：
    - `redis_url`
    - `redis_prefix`
- `werewolf_backend/requirements.txt`
  - 增加 Redis 客户端依赖
- 新增 `werewolf_backend/cache/`

### 推荐目录

```text
werewolf_backend/cache/
├── client.py          # Redis 客户端初始化
├── keys.py            # Redis key 规则
├── session_store.py   # 活跃对局读写
└── serializers.py     # 状态序列化/反序列化
```

### 本阶段要完成的内容

1. 建立 Redis 连接
2. 约定 key 命名规则
3. 实现 `game_id` 对应的活跃会话存取
4. 支持对局暂停/停止标志写入 Redis
5. 支持对局结束后清理或过期 Redis key

### 本阶段结果

完成后应达到：
- 活跃对局不再只依赖进程内内存
- Redis 成为运行时会话状态载体

---

## 阶段三：建立核心游戏状态模型

### 目标

把对局从“临时消息响应”升级为真正的对局状态机。

### 建议拆分

在 `werewolf_backend/game_engine/` 中建立至少以下模块：

- `models.py`
  - 玩家状态
  - 角色类型
  - 阶段类型
  - 对局状态
- `role_assignment.py`
  - 人数 -> 身份分配逻辑
- `state_machine.py`
  - 阶段推进逻辑
- `win_checker.py`
  - 胜负判定
- `engine.py`
  - 游戏主循环调度入口

### 本阶段要解决的问题

1. 如何表示一个完整对局
2. 如何表示玩家状态：
   - 存活
   - 死亡
   - 放逐
3. 如何表示阶段：
   - `night`
   - `day_speech`
   - `voting`
   - `finished`
4. 如何做人数与身份分配
5. 如何进行胜负判断
6. 如何与 Redis 状态序列化边界对齐

### 设计要求

这里应明确：
- `game_engine` 负责定义“规范状态”
- `cache/serializers.py` 负责把状态变成 Redis 可存格式

不要直接把随意 dict 当成长期状态协议。

---

## 阶段四：实现 WebSocket 会话调度与广播机制

### 目标

让 WebSocket 不只是一个 echo server，而是真正管理对局生命周期。

### 建议拆分

在 `werewolf_backend/websocket/` 中补充：

- `manager.py`
  - 管理连接
  - 广播事件
  - 管理进程内 socket 对象
- `schemas.py`
  - 定义 WebSocket 命令与事件格式
- `dispatcher.py`
  - 命令解析与分发

### 本阶段核心职责

1. 解析前端发来的 `cmd`
2. 根据 `game_id` 或当前会话查 Redis 中的运行态
3. 调用对应游戏逻辑
4. 把游戏引擎产生的事件广播给前端
5. 支持暂停 / 停止等控制流

### 设计建议

不要让 `websocket/routes.py` 承担全部逻辑。
这个文件只保留：
- 接入入口
- 调用 manager / dispatcher

## 阶段六：实现 SQLite 结构化日志与持久化

### 目标

满足课题中“可复盘、可评测、可观测”的要求。

### 推荐目录

- `werewolf_backend/database/`

### 推荐内容

- `models.py`

## 阶段九：接入人类玩家输入

### 目标

在纯 AI 跑通后，再把当前前端的人机交互正式接到后端。

### 当前前端已具备的能力

前端已经有：
- 人类发言输入
- 人类投票输入
- 人类技能输入
- 技能区按角色变化

但这些目前只是前端本地预览逻辑，还没有正式进入后端裁决。

### 后端后续需要支持的输入协议

建议增加以下命令之一：

#### 方案 A：统一命令

```json
{
  "cmd": "HUMAN_SKILL",
  "skill": "werewolf_kill",
  "target_id": 6
}
```

#### 方案 B：按角色拆分命令

```json
{"cmd": "HUMAN_SPEAK", "content": "..."}
{"cmd": "HUMAN_VOTE", "target_id": 4}
{"cmd": "HUMAN_WEREWOLF_KILL", "target_id": 6}
{"cmd": "HUMAN_SEER_CHECK", "target_id": 6}
{"cmd": "HUMAN_WITCH_HEAL"}
{"cmd": "HUMAN_WITCH_POISON", "target_id": 6}
{"cmd": "HUMAN_HUNTER_SHOOT", "target_id": 6}
```

### 接入顺序建议

1. 先支持 `HUMAN_SPEAK`
2. 再支持 `HUMAN_VOTE`
3. 最后支持不同角色技能

### 与 Redis 的关系

人类输入等待态建议存入 Redis，例如：
- 当前是否等待人类发言
- 当前是否等待人类投票
- 当前是否等待人类技能
- 当前可提交的目标范围摘要

因为这类状态属于活跃对局运行态，而不是历史归档数据。

## 8. 推荐第一个可交付版本（Version 0）

建议先定义一个非常明确的后端首个版本目标：

### Version 0 目标

当用户点击前端“开始”按钮时，后端能够：

1. 接收 `GAME_START`
2. 在 Redis 中创建活跃对局 session
3. 生成 6~12 人玩家数据
4. 自动分配身份
5. 返回 `GAME_STARTED`
6. 推送 `PHASE_CHANGE`
7. 推送几条 `AGENT_SPEAK`
8. 推送一次 `AGENT_STATUS_CHANGE`
9. 推送 `GAME_OVER`
10. 至少把一局的基础元数据和事件摘要写入 SQLite

### Version 0 的意义

它虽然还不是完整狼人杀，但已经能证明：

- 前后端协议是通的
- 前端展示链路是通的
- WebSocket 实时事件机制是通的
- Redis 会话层是通的
- SQLite 持久化链路是通的

这是最适合当前项目节奏的切入点。

## 9. Redis 设计注意事项

为了避免后端后面难以维护，建议从一开始就注意：

1. **不要把 Redis 用成历史数据库**
   - Redis 只保活跃态
   - 历史事件必须进 SQLite

2. **不要过度碎片化 key**
   - 早期不要把每个小字段拆成大量分散 key
   - 优先使用较稳定的 session 粒度

3. **不要存不可序列化对象**
   - WebSocket 连接、Python 对象实例不要直接写入 Redis

4. **状态序列化要有 schema**
   - Redis 中的数据结构不要临时拼接
   - 应与 `game_engine` 的模型边界一致

5. **结束态必须清理**
   - 对局完成后应删除或设置过期时间
   - 避免 Redis 堆积大量失效 session

---

## 10. 验证清单

每完成一个阶段，建议至少验证以下内容：

### 基础链路验证

1. `GET /` 可访问
2. `GET /api/health` 返回正常
3. WebSocket 能连接并收到 `CONNECTED`
4. Redis 连接成功

### 协议验证

5. `GAME_START` 能触发 `GAME_STARTED`
6. 能收到至少一个 `PHASE_CHANGE`
7. 能收到至少一个 `AGENT_SPEAK`
8. 能收到至少一个 `AGENT_STATUS_CHANGE`
9. 最终能收到 `GAME_OVER`

### Redis 验证

10. `GAME_START` 后 Redis 中存在活跃 session
11. `GAME_PAUSE` / `GAME_STOP` 能正确更新控制状态
12. 对局结束后 Redis key 被清理或设置过期

### SQLite 验证

13. 对局元数据成功落库
14. 关键事件成功落库
15. 即使 Redis 清空，历史结果仍可追溯

### 前端联调验证

16. 前端玩家卡片能成功初始化
17. 全局发言流能显示后端推送的发言
18. 玩家死亡/放逐状态能正确更新
19. 阶段与胜者信息能正确显示

### 后续系统验证

20. 一局完整对局能稳定跑完
21. 人类输入命令不会破坏 AI-only 主流程

## 11. 结论

后端下一步最合理的实现路径，不是从“最复杂的 Agent 智能”开始，而是按下面顺序推进：

1. 先对齐前端协议
2. 先搭 Redis 会话层
3. 再建立游戏状态机
4. 再补全规则结算
5. 再把关键事件落 SQLite
6. 最后接 Agent、LLM 与人机混战

这样可以最快形成一个：

- 可联调
- 可演示
- 可扩展
- 可缓存
- 可复盘

的后端主干。

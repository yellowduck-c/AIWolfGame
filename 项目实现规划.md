# AI狼人杀多智能体博弈项目开发需求文档

（适配 **Claude Code** 辅助开发 | 基于课题要求重新完善版）

## 一、项目背景与目标

本项目旨在基于 **Multi-Agent 协作框架**，构建一个能够自主完成 **信息不对称博弈** 的狼人杀 Agent Team 系统。

系统核心不只是“跑通狼人杀流程”，而是围绕 **多智能体协作 / 对抗 / 推理 / 决策机制** 展开设计与实现，使不同角色 Agent 在严格的信息边界下完成发言、行动、投票与博弈。

项目最终需要形成一个具备以下能力的完整系统：

1. 支持 6~12 人狼人杀对局的完整运行
2. 支持多个角色 Agent 基于各自目标自主行动
3. 支持信息隔离条件下的策略推理与阵营对抗
4. 支持对局过程全量记录与结构化观测
5. 支持前端观战界面，直观展示实时博弈过程
6. 支持纯 AI 对战，并为人机混战预留扩展能力

---

## 二、项目定位

这是一个 **AI 多智能体博弈系统项目**，目标是验证和展示以下能力：

- 多角色 Agent 的目标建模能力
- 多智能体之间的协作与对抗机制
- 严格信息不对称条件下的推理与决策能力
- 对局过程驱动、状态管理与胜负裁决能力
- LLM 在复杂规则博弈场景中的行为生成能力
- 对局日志、行为轨迹、结果复盘与后续评测能力

因此，项目实现重点应放在：

- **Agent 设计是否合理**
- **游戏引擎是否完整稳定**
- **信息隔离是否严格执行**
- **行为日志是否足够结构化、可复盘、可评测**
- **前端是否能清晰呈现多 Agent 实时博弈过程**

## 三、核心需求

### 1. 基础对局需求

1. 支持 **6~12 人** 对局
2. 前端负责人数输入与校验
3. 后端根据总人数自动完成身份分配
4. 身份配比规则：
   - 狼人 = 总人数 // 3
   - 固定神职：预言家、女巫、猎人
   - 其余为平民
5. 系统需支持完整回合制流程推进

### 2. 标准游戏流程需求

采用简化但完整的狼人杀流程：

1. 创建对局
2. 分配身份
3. 游戏开始
4. 黑夜阶段
   - 狼人行动
   - 预言家查验
   - 女巫决策
5. 白天阶段
   - 公布夜晚结果
   - 玩家依次发言
   - 投票放逐
   - 处理死亡/出局状态
6. 胜负判定
7. 游戏结束并输出结果

系统必须具备稳定的阶段切换机制、状态更新机制与终局判断机制。

### 3. 多智能体 Agent 核心需求

每个 Agent 应基于统一框架构建，但能够因角色不同而表现出差异化目标与行为。

必须满足：

1. **统一 Agent 基类**
   - 定义公共输入输出
   - 定义行为接口，如发言、投票、技能释放、状态更新
   - 定义记忆读写接口

2. **角色差异化能力**
   - 狼人：协作、隐藏身份、诱导投票、夜间击杀
   - 预言家：查验身份、建立可信推理链
   - 女巫：使用解药/毒药并平衡收益风险
   - 猎人：在死亡条件下执行开枪策略
   - 平民：基于公开信息进行推理、发言与投票

3. **多层记忆机制**
   - 短期记忆：当前局内最近事件、近期发言、最近推理
   - 长期记忆：可复用的策略经验、偏好、对局统计
   - 私有记忆：角色独享信息、夜间结果、身份判断

4. **行为生成能力**
   以下行为应由大模型或 Agent 策略模块驱动：
   - 发言
   - 投票
   - 技能决策
   - 阵营内协作决策
   - 回合内推理与目标选择

### 4. 信息隔离与博弈约束

本项目的关键要求之一是 **严格信息隔离**。

必须明确：

1. 每个 Agent 只能看到其“规则上应当知道”的信息
2. 好人阵营不能直接获知狼人身份
3. 非狼人角色不能访问狼人夜间协作信息
4. 每个角色只能访问与自身能力相关的私有信息
5. 公共发言、放逐结果、死亡公告等属于公共信息
6. Agent 的内部推理、记忆和中间决策默认不可被其他 Agent 直接读取

系统设计上需要把：

- 公共信息流
- 阵营共享信息流
- 私有角色信息流

明确区分，防止出现越权读取或信息泄漏。

### 5. 对局引擎需求

系统需搭建完整的对局引擎，负责驱动整场游戏运行。

游戏引擎至少应承担以下职责：

1. 创建并初始化对局
2. 生成人员与身份映射
3. 管理回合/阶段流转
4. 调度各角色 Agent 在正确时机行动
5. 管理玩家状态（存活、死亡、放逐等）
6. 处理投票、技能、结算逻辑
7. 执行胜负判定
8. 输出完整过程日志和最终结果

### 6. 可观测性与结构化日志需求

系统必须输出 **结构化日志**，实现全程可观测。

至少应支持记录以下内容：

1. 对局基础信息
   - 对局 ID
   - 玩家数量
   - 身份分配结果
   - 模型配置 / Agent 配置

2. 阶段级事件
   - 当前阶段
   - 阶段开始/结束时间
   - 阶段内关键决策

3. Agent 行为事件
   - 发言内容
   - 投票对象
   - 技能目标
   - 状态变化

4. 结果事件
   - 夜间死亡结果
   - 放逐结果
   - 胜负结果
   - 存活阵营统计

5. 可扩展记录
   - 推理摘要
   - 行为理由摘要
   - 复盘标签
   - 评测指标结果

日志应尽量以结构化方式保存，便于：

- 前端实时展示
- 对局回放
- 结果评测
- 复盘分析
- 不同模型/版本横向对比

## 四、系统形态与使用场景

### 1. 运行模式

系统至少支持以下模式：

1. **纯 AI 对战**
   - 所有角色均由 Agent 控制
   - 用于验证多智能体博弈能力

2. **人机混战（扩展模式）**
   - 保留人工玩家接入能力
   - 人类玩家可以替代部分角色参与对局
   - 系统仍需保证整体流程完整推进

### 2. 前端观战 UI（重要加分项）

前端需提供观战与控制能力，至少覆盖以下目标：

1. 创建并启动对局
2. 观察所有 Agent 的实时状态变化
3. 查看每个 Agent 的发言过程
4. 查看全局对局日志与关键事件
5. 在上帝视角下直观理解整场博弈推进

如果实现人机混战，则前端还应进一步支持：

- 人类玩家交互输入
- 等待人工操作的流程控制
- 人机共存下的状态同步

## 五、总体架构设计

推荐采用前后端分离架构：

```text
前端 Vue3
  ├─ 对局创建 / 控制
  ├─ 上帝视角观战 UI
  └─ WebSocket 实时展示

后端 FastAPI
  ├─ HTTP API
  ├─ WebSocket 实时通信
  ├─ 游戏引擎
  ├─ Agent Team 调度层
  ├─ LLM 服务层
  └─ 日志/存储层
```

通信方式：

- **HTTP**：创建对局、读取历史结果、初始化配置
- **WebSocket**：推送实时状态、阶段事件、发言内容、控制反馈

## 八、前端展示需求（观战 UI）

游戏开始后，前端应以 **上帝视角** 实时展示整场博弈过程。

### 1. Agent 信息面板

每个 Agent 卡片至少展示：

- 玩家 ID
- 真实角色身份
- 阵营
- 当前状态（存活 / 死亡 / 放逐）
- 当前发言
- 历史发言摘要或最近发言记录

### 2. 全局实时发言流

应展示：

- 时间戳
- 玩家 ID
- 角色
- 发言内容

并支持实时追加、滚动查看。

### 3. 状态变化展示

至少包括：

- 死亡高亮
- 放逐标记
- 当前发言人高亮
- 阶段切换提示
- 游戏结束结果展示

### 4. 布局要求

- 6~8 人优先单行或紧凑布局
- 9~12 人采用双行网格
- 卡片不得重叠或明显错位
- 死亡 / 放逐玩家应明显灰化或特殊标记

### 5. 控制能力

前端至少提供：

- 开始游戏
- 暂停游戏
- 结束游戏
- 调整播放 / 推进速度（如实现）

## 九、通信协议建议

### 1. WebSocket：前端 → 后端

```json
{"cmd":"GAME_START"}
{"cmd":"GAME_PAUSE"}
{"cmd":"GAME_STOP"}
```

如支持扩展，可继续加入：

```json
{"cmd":"SET_SPEED","value":2}
{"cmd":"HUMAN_ACTION","playerId":3,"action":{}}
```

### 2. WebSocket：后端 → 前端

#### 游戏开始

```json
{
  "event": "GAME_STARTED",
  "agents": [
    { "id": 1, "role": "狼人", "camp": "狼人", "status": "alive" },
    { "id": 2, "role": "预言家", "camp": "好人", "status": "alive" }
  ]
}
```

#### Agent 发言

```json
{
  "event": "AGENT_SPEAK",
  "id": 1,
  "role": "狼人",
  "content": "我是平民，这轮投4号"
}
```

#### 状态变化

```json
{
  "event": "AGENT_STATUS_CHANGE",
  "id": 4,
  "status": "dead"
}
```

#### 阶段切换

```json
{
  "event": "PHASE_CHANGE",
  "phase": "day_speech"
}
```

#### 游戏结束

```json
{
  "event": "GAME_OVER",
  "winner": "好人阵营"
}
```

## 十、非功能性要求

### 1. 稳定性

- 对局流程不可因单次消息异常直接中断
- 状态切换必须可追踪、可回放
- 多个 Agent 的行动顺序必须可控

### 2. 可扩展性

系统应为以下扩展预留空间：

- 新角色接入
- 新模型接入
- 人类玩家接入
- 复盘模块接入
- 评测模块接入
- Leaderboard 接入

### 3. 可维护性

- Agent 层、游戏引擎层、通信层、展示层职责应清晰分离
- 日志字段应保持稳定，便于后续评测与回放

## 十二、进阶方向（三选一）

### 方向一：通用 Agent

探索“读懂自己 → 修改自己 → 运行自己”的自演化系统，使通用 Agent 逐步演化为适用于狼人杀多角色博弈的专用 Agent。

### 方向二：评测 + 复盘

构建多维可量化评测体系，包括但不限于：

- 结果评测
- 过程评测
- 发言质量评测
- 推理有效性评测
- 不同模型 / 不同版本 Agent 同台竞技

最终支持形成对局复盘归因与 Leaderboard。

### 方向三：自进化 Agent

实现“对局 → 分析 → 优化 → 再对局”的闭环，使不同角色 Agent 在多局博弈中持续提升策略与胜率。

## 十三、最终交付效果

项目完成后，理想交付效果应为：

1. 用户可创建 6~12 人狼人杀对局
2. 系统自动分配身份并启动完整对局
3. 多个角色 Agent 在信息隔离条件下自主行动
4. 系统可实时展示发言、投票、状态变化与胜负结果
5. 全过程具备结构化日志，可用于回放、复盘、评测
6. 支持纯 AI 对战，并具备扩展到人机混战的能力
7. 形成一个可持续扩展的多智能体博弈实验平台

# 项目提示词设计分析

## 1. 设计目标

该项目的提示词系统并不是单一的“一个总 prompt 跑全流程”，而是一个**按行动类型拆分、按角色注入特化信息、按当前局面动态构造输入**的提示词体系。

它的设计目标主要有四个：

1. **让同一角色在不同动作阶段使用不同提示词约束**。
2. **让不同角色共享一套框架，但拥有不同的行为指导**。
3. **把当前游戏状态和历史记忆结构化注入模型，降低自由发挥带来的越权和幻觉**。
4. **强约束输出格式，方便后端解析成行为决策。**

当前提示词设计主要分布在：

- `werewolf_backend/llm_service/prompt_builder.py`
- `werewolf_backend/agent/roles/role_profiles.py`
- `werewolf_backend/agent/roles/base_agent.py`
- `werewolf_backend/agent/state/schemas.py`
- `werewolf_backend/llm_service/client.py`

---

## 2. 整体架构：提示词不是硬编码一段文本，而是动态拼装

系统并不是把所有规则直接写死在一个 prompt 字符串里，而是分成三层：

### 2.1 基础层：通用行为约束

位置：[prompt_builder.py:13](werewolf_backend/llm_service/prompt_builder.py#L13)

```python
PROMPT_BASE_INSTRUCTION = "你是狼人杀游戏中的一名玩家。只基于提供给你的信息做决策，不要编造额外规则，不要输出分析过程。"
```

这是一条全局底线约束，所有动作共用。它主要解决三个问题：

- 限制模型只基于输入信息行动
- 防止模型自己发明额外规则
- 不允许输出思维链或分析过程

这是整个 prompt 体系的“系统人格边界”。

### 2.2 行动层：按动作类型给不同输出任务

位置：[prompt_builder.py:15-38](werewolf_backend/llm_service/prompt_builder.py#L15-L38)

系统把动作拆成四类：

- `speech`
- `camp_chat`
- `vote`
- `skill`

每类动作都定义了自己的指令模板。例如：

#### speech
- 目标：生成一句自然中文发言
- 输出：`{"content":"..."}`

#### camp_chat
- 目标：生成一句狼人夜间私聊
- 限制：只能基于 `public_state / private_state / camp_shared_state / legal_actions`
- 输出：`{"content":"..."}`

#### vote
- 目标：从候选人中选出投票对象
- 输出：`{"target_id":2}`

#### skill
- 目标：使用角色技能
- 限制：只能用 `legal_actions` 允许的 skill 和 target
- 若是狼人，若私聊中有明确统一目标，应优先服从
- 输出：`{"skill":"inspect","target_id":3}`

这说明该系统不是“让模型自由聊天”，而是把每一步都包装成**结构化决策任务**。

### 2.3 角色层：通过 specialization 注入角色画像与行为指引

提示词的真正差异化主要来自 `role_profiles.py` 和 `base_agent.py`。

## 3. 角色提示词是如何注入的

### 3.1 RoleProfile：角色画像

位置：[role_profiles.py:25-46](werewolf_backend/agent/roles/role_profiles.py#L25-L46)

`RoleProfile` 定义了一个角色的静态行为画像：

- `role`
- `camp`
- `identity`
- `objective`
- `disclosure_policy`
- `cooperation_style`
- `behavior_tags`
- 各行动的 guidance

例如狼人画像位置：[role_profiles.py:49-81](werewolf_backend/agent/roles/role_profiles.py#L49-L81)

狼人画像强调：

- 夜间协作、白天伪装
- 不暴露狼人身份
- 参考共享信息协作
- speech / vote / skill / camp_chat 各自有不同目标与优先级

这意味着项目中的“角色提示词”不是一条长 prompt，而是一组可编排的元数据。

### 3.2 RoleActionGuidance：动作级指导

位置：[role_profiles.py:9-22](werewolf_backend/agent/roles/role_profiles.py#L9-L22)

每个动作指导包含四个维度：

- `goal`
- `constraints`
- `priorities`
- `style`

这非常重要，因为它把提示词从自然语言散文拆成了**可组合的结构化控制变量**。

例如狼人 `skill_guidance` 当前强调：

- 优先依据狼人私聊共识选择袭击目标
- 私聊已形成共同目标时不要无故分票
- 若没有共识，再选高价值好人

这就是这次修复后新增的强约束来源。

## 4. 提示词装配流程

### 4.1 enrich_decision_input 是提示词装配入口

位置：[base_agent.py:76-103](werewolf_backend/agent/roles/base_agent.py#L76-L103)

每次 agent 行动前，都会调用：

```python
enrich_decision_input(action_type, decision_input)
```

它会做几件事：

1. 根据角色获取 `RoleProfile`
2. 根据 action_type 选择该动作的 `RoleActionGuidance`
3. 构造 `specialization` 字段

注入内容包括：

- `role_profile`
- `action_type`
- `action_guidance`
- `behavior_tags`
- `role_instruction`
- `action_guidance_instruction`

这里很关键：

- **原始决策上下文** 负责提供状态数据
- **specialization** 负责提供行为解释框架

也就是说，项目把“世界状态”和“角色行为策略”分开建模。

### 4.2 role_instruction 与 action_guidance_instruction

位置：
- [base_agent.py:52-61](werewolf_backend/agent/roles/base_agent.py#L52-L61)
- [base_agent.py:63-71](werewolf_backend/agent/roles/base_agent.py#L63-L71)

项目并不是直接把 dataclass 原样塞给模型，而是把它们转成可读指令文本：

#### role_instruction
示例结构：
- 你的身份定位：...
- 你的阵营目标：...
- 你的信息公开原则：...
- 你的协作风格：...
- 行为标签：...

#### action_guidance_instruction
示例结构：
- 当前行动目标：...
- 行动约束：...
- 行动优先级：...
- 表达/决策风格：...

这一步的意义在于：

- 结构化数据方便程序维护
- 文本化指令方便 LLM 理解

## 5. Prompt Builder 的拼装方式

### 5.1 统一 payload 注入

位置：[prompt_builder.py:40-58](werewolf_backend/llm_service/prompt_builder.py#L40-L58)

`_dump_payload()` 会把 `AgentDecisionInput` 序列化成 JSON 文本，包含：

- agent_id
- role
- camp
- phase
- round
- public_state
- private_state
- camp_shared_state
- memory_summary
- legal_actions
- specialization

也就是说，模型拿到的不是零碎变量，而是一整份完整的结构化状态快照。

### 5.2 _build_prompt 的模板框架

位置：[prompt_builder.py:96-105](werewolf_backend/llm_service/prompt_builder.py#L96-L105)

普通 prompt 由以下部分拼接：

1. `PROMPT_BASE_INSTRUCTION`
2. `role_instruction`
3. `action_guidance_instruction`
4. 当前动作的 `ACTION_INSTRUCTIONS[action_type]`
5. `输入：{payload}`

所以它的统一抽象可以写成：

```text
通用约束
+ 角色定位
+ 当前动作指导
+ 本动作专属要求
+ 当前结构化输入
```

这是一个典型的“层叠式 prompt 设计”。

## 6. 四类动作提示词的设计思路

### 6.1 发言 prompt（speech）

位置：[prompt_builder.py:16-19](werewolf_backend/llm_service/prompt_builder.py#L16-L19)

特点：

- 目标是自然发言
- 输出结构是 `content`
- 强调简短中文

它不是纯聊天，而是“角色扮演 + 结构化输出”。

#### 特殊点：speech 使用了可选模板对象

位置：[prompt_builder.py:108-145](werewolf_backend/llm_service/prompt_builder.py#L108-L145)

如果环境中安装了 `ChatPromptTemplate`，speech 会优先走模板化链路：

- system message：通用底线
- human message：角色指导 + 动作指导 + payload

说明开发者认为 speech 是最接近自然语言生成的动作，因此给了更细的消息结构。

### 6.2 狼人私聊 prompt（camp_chat）

位置：[prompt_builder.py:20-25](werewolf_backend/llm_service/prompt_builder.py#L20-L25)

特点：

- 明确告诉模型这是“夜间私聊”而非公开发言
- 明确限定只能基于当前可见信息
- 输出仍然是结构化 JSON

该 prompt 的作用不是做最终决策，而是帮助狼人阵营形成局部共识。

### 6.3 投票 prompt（vote）

位置：[prompt_builder.py:26-30](werewolf_backend/llm_service/prompt_builder.py#L26-L30)

特点：

- 非常短
- 约束极强
- 只要求从候选集中选一个 id

它本质上是一个“受限分类任务”。

### 6.4 技能 prompt（skill）

位置：[prompt_builder.py:31-36](werewolf_backend/llm_service/prompt_builder.py#L31-L36)

这是目前最复杂的一类。

它不仅要求：

- 技能合法
- 目标合法
- 必要时支持 `target_id = null`

还加入了一个狼人专属约束：

- 如果 `camp_chat_history` 已形成明确共同目标，应优先执行

这说明 skill prompt 已经开始承担“将阵营协作文本转化为执行决策”的桥梁作用。

## 7. 输出格式设计：为什么全部要求 JSON

项目中的所有 LLM 输出都尽量要求 JSON：

- speech / camp_chat → `{"content":"..."}`
- vote → `{"target_id":2}`
- skill → `{"skill":"...","target_id":...}`

原因很明确：

1. **便于程序解析**
2. **降低自然语言歧义**
3. **便于 fallback 校验**
4. **可以直接映射到 dataclass**

对应解析逻辑在 [client.py](werewolf_backend/llm_service/client.py) 中：

- `_parse_json_object()`
- `_parse_speech_decision()`
- `_parse_vote_decision()`
- `_parse_skill_decision()`
- `_parse_text_content()`

这套解析器说明提示词设计和输出解析是联动设计的，而不是各自独立。

---

## 8. 提示词系统与运行时校验是双保险关系

项目并不完全相信 prompt 会百分百生效。

它采取的是：

- **prompt 前置约束**：告诉模型该怎么做
- **程序后置校验**：如果输出不合法就拒绝或 fallback

例如技能决策在 `client.py` 中会调用 `_is_valid_skill_choice()` 做检查：

- 是否 allowed
- skill 是否匹配角色合法技能
- target 是否在 legal targets 中
- 女巫 heal / poison 是否满足额外规则
- 猎人是否满足触发条件

也就是说，这套提示词不是“全权交给模型”，而是“**模型负责在约束空间内给出候选，后端负责最终合法性把关**”。

## 9. 当前提示词设计的优点

### 9.1 模块化强

- action instruction 独立
- role profile 独立
- prompt builder 独立
- 决策上下文独立

后续要新增角色或动作，不必重写整个系统。

### 9.2 角色差异与动作差异都可表达

同一个狼人：

- 白天发言像村民
- 夜晚私聊偏协作
- 夜晚刀人偏执行

这三种行为由不同 guidance 分别控制。

### 9.3 和信息隔离机制高度一致

prompt 里明确引用：

- `public_state`
- `private_state`
- `camp_shared_state`
- `legal_actions`

这让提示词天然受后端状态权限模型约束。

### 9.4 易于做定向调优

这次“狼人明明私聊统一目标却仍分票”的修复就是典型案例：

- 不需要推翻整套 prompt
- 只需要增强狼人 `skill_guidance`
- 再增强 `ACTION_INSTRUCTIONS["skill"]`

改动面小，但行为影响明显。

## 10. 当前提示词设计的局限

### 10.1 共识仍依赖模型自己从文本里理解

即使 `camp_chat_history` 进入了 prompt，模型仍需自行判断：

- 有没有共识
- 哪条是最新共识
- 是否应该服从

这会导致不稳定。

### 10.2 payload 体积会不断增长

由于：

- `public_events` 线性增长
- `memory_summary` 线性增长
- `camp_chat_history` 线性增长

prompt 会越来越长，后期可能影响质量与成本。

### 10.3 speech 使用模板，其他动作仍是 plain text 拼接

目前只有 speech 走了 `ChatPromptTemplate` 分层消息模板，其它动作仍是单段文本拼装。长期看可以统一成更规范的消息模板。

### 10.4 角色 guidance 仍是静态规则集合

当前 guidance 虽然结构化，但仍偏静态。它缺少：

- 局势阶段感知的动态策略
- 发言可信度建模
- 共识强度建模
- 更细粒度的冲突解决规则

---

## 11. 可以如何继续演进

### 11.1 增加“共识提取器”而不是只喂聊天记录

不要只把 `camp_chat_history` 原文给模型。
可以在后端先额外生成：

- `consensus_target_id`
- `consensus_confidence`
- `consensus_reason`

再进入 skill prompt。

### 11.2 给不同动作统一用 ChatPromptTemplate

把 `camp_chat`、`vote`、`skill` 也模板化，便于：

- 统一调参
- 更清晰区分 system/human 指令
- 更方便 A/B 测试

### 11.3 引入阶段化 prompt 策略

例如狼人：

- 第 1 夜更偏身份猜测
- 中盘更偏收益最大化
- 残局更偏自保与控票

### 11.4 对长历史做摘要再注入

将 `public_events`、`memory_summary`、`camp_chat_history` 做压缩摘要，以提升模型聚焦度。

---

## 12. 结论

本项目的提示词设计本质上是一套**结构化、模块化、动作驱动、角色特化**的决策提示系统。

它的核心链路可以概括为：

1. 后端构造 `AgentDecisionInput`
2. BaseAgent 根据角色和动作注入 `specialization`
3. PromptBuilder 将通用约束、角色定位、动作指导、结构化 payload 拼成 prompt
4. LLM 输出 JSON 决策
5. 后端解析并做合法性校验

这套设计的优点不在于 prompt 文案多华丽，而在于它把：

- 状态
- 权限
- 角色目标
- 动作约束
- 输出结构

统一成了一条清晰的可编排链路。

如果从分析角度看，这个项目的提示词系统已经具备了一个成熟 agent runtime 的基本雏形：**不是“问模型一句话”，而是在“给模型一个被精确定义的行动问题”**。

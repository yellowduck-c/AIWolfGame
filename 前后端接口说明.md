# 前后端接口说明

本文档用于说明当前 `werewolf_frontend` 与 `werewolf_backend` 之间已经存在、以及前端已经约定好的接口，方便后续补全后端实现。

## 1. 总览

当前项目的前后端通信分为两类：

- **HTTP**：用于基础健康检查，以及后续可扩展的对局创建、配置读取、历史结果查询
- **WebSocket**：用于对局实时控制与事件推送

当前代码中：

- 前端实际已经接入并使用了 **1 个 HTTP 基础地址** 与 **1 个 WebSocket 连接地址**
- 后端当前已实现：
  - `GET /`
  - `GET /api/health`
  - `WS /ws`
- 前端已经约定了一批对局控制命令与事件消息格式，后端后续应按这些格式实现

---

## 2. 环境变量与基础地址

### 前端读取方式

文件：`werewolf_frontend/src/utils/websocket.ts`

```ts
export const apiBaseUrl = import.meta.env.VITE_API_BASE_URL ?? 'http://127.0.0.1:8000'
export const wsBaseUrl = import.meta.env.VITE_WS_BASE_URL ?? 'ws://127.0.0.1:8000/ws'
```

### 当前约定

- HTTP 基础地址：`VITE_API_BASE_URL`
- WebSocket 地址：`VITE_WS_BASE_URL`

### 当前本地默认示例

通常本地运行时会配置为：

- HTTP：`http://127.0.0.1:8010`
- WebSocket：`ws://127.0.0.1:8010/ws`

## 3. 已实现 HTTP 接口

## 3.1 根路径

### 请求

- **Method**: `GET`
- **Path**: `/`

### 后端位置

文件：`werewolf_backend/main.py`

### 当前返回

```json
{
  "message": "AI Werewolf backend is running"
}
```

### 作用

- 用于确认后端服务已启动
- 前端当前未直接使用该接口

---

## 3.2 健康检查接口

### 请求

- **Method**: `GET`
- **Path**: `/api/health`

### 后端位置

文件：`werewolf_backend/api/routes.py`

### 当前返回

```json
{
  "status": "ok",
  "app": "<app_name>",
  "port": 8010
}
```

其中：

- `status`: 固定为 `ok`
- `app`: 后端应用名
- `port`: 当前运行端口

### 作用

- 用于确认后端 API 正常响应
- 可作为前端后续启动前探活接口

### 请求

### 后端位置

## 4.2 当前后端已实现的 WebSocket 行为

### 连接成功后，后端主动发送

```json
{
  "event": "CONNECTED",
  "message": "WebSocket ready"
}
```

### 前端处理方式

文件：`werewolf_frontend/src/utils/websocket.ts`

```ts
case 'CONNECTED':
  gameStore.setConnectionStatus('connected', event.message)
  break
```

### 作用

- 告知前端 WebSocket 已建立
- 更新前端连接状态展示

---

## 4.3 当前后端已实现的回显行为

后端当前是一个占位实现：

- 前端发送任意文本
- 后端原样回显为 `ECHO` 事件

### 后端返回格式

```json
{
  "event": "ECHO",
  "payload": "<原始文本>"
}
```

### 前端处理方式

```ts
case 'ECHO':
  gameStore.setConnectionStatus('connected', `后端已收到指令：${event.payload}`)
  break
```

### 作用

- 当前仅用于验证链路通畅
- 后续应替换为真实对局控制逻辑

## 5. 前端已发送的 WebSocket 命令

当前前端已经固定使用以下命令控制对局。

文件：`werewolf_frontend/src/utils/websocket.ts`

```ts
export function sendGameCommand(command: 'GAME_START' | 'GAME_PAUSE' | 'GAME_STOP'): void {
  socket.send(JSON.stringify({ cmd: command }))
}
```

文件：`werewolf_frontend/src/views/GameBattle.vue`

- `sendGameCommand('GAME_START')`
- `sendGameCommand('GAME_PAUSE')`
- `sendGameCommand('GAME_STOP')`

因此，后端应支持以下前端 -> 后端 WebSocket 消息：

```json
{
  "cmd": "GAME_START"
}
```

- 前端控制栏“开始”按钮

### 后端建议行为

- 初始化对局
- 分配角色
- 推送 `GAME_STARTED`
- 推送后续阶段事件

---

## 5.2 暂停对局

```json
{
  "cmd": "GAME_PAUSE"
}
```

### 触发位置

- 前端控制栏“暂停”按钮

### 后端建议行为

- 暂停当前调度
- 可推送状态确认事件（当前前端尚未单独定义）

---

## 5.3 结束对局

```json
{
  "cmd": "GAME_STOP"
}
```

### 触发位置

- 前端控制栏“结束”按钮

### 后端建议行为

- 终止当前对局

- 可直接推送 `GAME_OVER` 或单独的停止确认事件

## 6. 前端已约定的后端 -> 前端 WebSocket 事件

这些事件已经在前端类型与处理逻辑中写好，后端实现时应尽量按该格式输出。

文件：`werewolf_frontend/src/utils/websocket.ts`

```ts
type BackendEvent =
  | { event: 'GAME_STARTED'; agents: Array<{ id: number; role: string; camp: string; status: 'alive' | 'dead' | 'exiled' }> }
  | { event: 'AGENT_SPEAK'; id: number; role: string; content: string }
  | { event: 'AGENT_STATUS_CHANGE'; id: number; status: 'alive' | 'dead' | 'exiled' }
  | { event: 'PHASE_CHANGE'; phase: string }
  | { event: 'GAME_OVER'; winner: string }
  | { event: 'CONNECTED'; message: string }
  | { event: 'ECHO'; payload: string }
```

下面按事件分别说明。

---

## 6.1 `GAME_STARTED`

### 数据格式

```json
{
  "event": "GAME_STARTED",
  "agents": [
    {
      "id": 1,
      "role": "狼人",
      "camp": "狼人",
      "status": "alive"
    }
  ]
}
```

### 字段说明

- `id`: 玩家编号
- `role`: 玩家真实身份（前端当前会直接接收）
- `camp`: 玩家阵营
- `status`: 玩家状态，枚举值：
  - `alive`
  - `dead`
  - `exiled`

### 前端处理

```ts
case 'GAME_STARTED':
  gameStore.initializeAgents(event.agents)
  gameStore.setPhase('night')
  break
```

### 作用

- 初始化前端玩家列表
- 进入对局阶段展示

### 注意

当前前端 `initializeAgents()` 默认把后端传入的玩家都标记为 AI；如果后端之后要正式支持人机混战，需要再扩展初始化协议，加入 `playerType` 或单独的人类玩家标识。

---

## 6.2 `AGENT_SPEAK`

### 数据格式

```json
{
  "event": "AGENT_SPEAK",
  "id": 3,
  "role": "预言家",
  "content": "我昨晚查验了 5 号。"
}
```

### 字段说明

- `id`: 发言玩家编号
- `role`: 发言玩家角色
- `content`: 发言文本

### 前端处理

```ts
case 'AGENT_SPEAK':
  gameStore.appendSpeech(event)
  break
```

### 作用

- 更新对应玩家卡片的“当前发言”
- 把发言写入全局发言流
- 更新最近发言历史

---

## 6.3 `AGENT_STATUS_CHANGE`

### 数据格式

```json
{
  "event": "AGENT_STATUS_CHANGE",
  "id": 5,
  "status": "dead"
}
```

### 字段说明

- `id`: 玩家编号
- `status`: 新状态
  - `alive`
  - `dead`
  - `exiled`

### 前端处理

```ts
case 'AGENT_STATUS_CHANGE':
  gameStore.updateAgentStatus(event)
  break
```

### 作用

- 更新玩家卡片生死状态
- 触发卡片视觉变化

---

## 6.4 `PHASE_CHANGE`

### 数据格式

```json
{
  "event": "PHASE_CHANGE",
  "phase": "day_speech"
}
```

### 当前前端已识别阶段

文件：`werewolf_frontend/src/views/GameBattle.vue`

```ts
const labels: Record<string, string> = {
  night: '夜晚行动',
  day_speech: '白天发言',
  voting: '投票放逐',
  finished: '对局结束',
}
```

### 建议后端优先使用的阶段值

- `night`
- `day_speech`
- `voting`
- `finished`

### 前端处理

```ts
case 'PHASE_CHANGE':
  gameStore.setPhase(event.phase)
  break
```

---

## 6.5 `GAME_OVER`

### 数据格式

```json
{
  "event": "GAME_OVER",
  "winner": "好人"
}
```

### 字段说明

- `winner`: 获胜阵营或获胜描述

### 前端处理

```ts
case 'GAME_OVER':
  gameStore.setWinner(event.winner)
  break
```

### 作用

- 更新顶部控制栏的胜者展示
- 标记对局结束

---

## 6.6 `CONNECTED`

### 数据格式

```json
{
  "event": "CONNECTED",
  "message": "WebSocket ready"
}
```

### 作用

- 当前用于连接建立确认

---

## 6.7 `ECHO`

### 数据格式

```json
{
  "event": "ECHO",
  "payload": "{\"cmd\":\"GAME_START\"}"
}
```

### 作用

- 当前仅用于开发阶段链路验证
- 正式后端完成后可删除或不再使用

## 7. 前端当前存在但尚未接入后端的“人类玩家操作”

当前前端在人机混战模式下，已经实现了本地交互面板，但这些操作**还没有通过 WebSocket/HTTP 发送给真实后端**，目前只是更新前端本地状态与日志。

相关文件：

- `werewolf_frontend/src/views/GameBattle.vue`
- `werewolf_frontend/src/stores/game_store.ts`

## 7.1 人类发言

当前前端行为：

- 输入文本
- 调用 `submitHumanSpeech()`
- 本地追加到发言流

### 当前尚未形成正式后端接口

建议后续补一个 WebSocket 消息，例如：

```json
{
  "cmd": "HUMAN_SPEAK",
  "content": "我先听 4 号发言。"
}
```

---

## 7.2 人类投票

当前前端行为：

- 选择目标玩家
- 调用 `submitHumanVote()`

## 8. 后端下一步实现建议

如果你马上开始补后端代码，建议优先按下面顺序完成：

1. **保持当前已实现接口不变**
   - `GET /`
   - `GET /api/health`
   - `WS /ws`

2. **在 `/ws` 内先支持 3 个基础控制命令**
   - `GAME_START`
   - `GAME_PAUSE`
   - `GAME_STOP`

3. **优先实现 5 个核心事件推送**
   - `GAME_STARTED`
   - `AGENT_SPEAK`
   - `AGENT_STATUS_CHANGE`
   - `PHASE_CHANGE`
   - `GAME_OVER`

4. **后续再扩展人类玩家输入协议**
   - `HUMAN_SPEAK`
   - `HUMAN_VOTE`
   - `HUMAN_SKILL` 或按角色拆分的技能命令

5. **如果要正式支持人机混战初始化**
   建议在 `GAME_STARTED` 事件里增加：
   - `playerType`
   - 或 `humanPlayerId`
   - 或更完整的身份可见性字段

  - 命令发送
  - 后端事件类型定义
  - 事件处理逻辑

### 后端

- `werewolf_backend/main.py`
  - 注册 HTTP / WebSocket 路由

- `werewolf_backend/api/routes.py`
  - `GET /api/health`

- `werewolf_backend/websocket/routes.py`
  - `WS /ws`
  - 当前 `CONNECTED` / `ECHO` 占位实现

---

## 10. 结论

目前项目已经形成了一个**明确的前端协议骨架**：

- 控制命令从前端发给后端
- 对局事件由后端持续推给前端
- 玩家状态、发言、阶段、结局都有固定的前端消费方式

因此，后端下一步只需要围绕这些既有协议补齐真实逻辑，就可以快速接上当前前端页面。

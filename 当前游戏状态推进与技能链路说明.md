## 2. 游戏会话与基础状态

### 2.1 GameSession 关键字段

当前游戏会话包含以下核心字段：

- `game_id`：对局 ID
- `phase`：当前阶段
  - `night`
  - `day_speech`
  - `voting`
  - `finished`
- `round`：当前回合数，从 1 开始
- `status`：运行状态
  - `running`
  - `paused`
  - `finished`
- `agents`：玩家快照列表
- `winner`：胜利阵营，未结束时为 `None`
- `public_events`：已公开事件流

玩家当前有三种状态：

- `alive`：存活
- `dead`：死亡
- `exiled`：被放逐但仍保留特殊显示身份

定义见：

## 3. 状态机与阶段推进机制

## 3.1 初始状态

新对局创建后：

- 初始阶段固定为 `night`
- 初始回合固定为 `1`
- 初始运行状态为 `running`

见：
- [werewolf_backend/game_engine/state_machine.py:11-20](werewolf_backend/game_engine/state_machine.py#L11-L20)

## 3.2 标准阶段顺序

当前标准阶段顺序为：

1. `night`
2. `day_speech`
3. `voting`
4. 再回到下一轮 `night`

定义见：
- [werewolf_backend/game_engine/state_machine.py:7](werewolf_backend/game_engine/state_machine.py#L7)
- [werewolf_backend/game_engine/state_machine.py:22-32](werewolf_backend/game_engine/state_machine.py#L22-L32)

具体推进规则：

- `night -> day_speech`
- `day_speech -> voting`
- `voting -> night`，并且 `round + 1`，但前提是投票阶段结束后游戏仍未结束

这意味着：

- 回合数只会在投票阶段结束且对局继续时，于重新进入夜晚时递增
- 如果投票阶段已经直接判定胜负，则 `phase` 会进入 `finished`，不会再推进到下一轮 `night`，也不会额外 `round + 1`
- 一个“完整白天轮次”包含：夜晚结算、白天发言、白天投票

## 3.3 执行器如何推进游戏

当前总执行入口是 `PhaseDrivenExecutor`。

其职责是：

- 根据当前 `phase` 分发到对应 action
- 在阶段执行后推进到下一阶段
- 在夜晚后和投票后做胜负检查
- 在游戏结束时停止迭代

核心入口见：
- [werewolf_backend/game_engine/executor.py:16-162](werewolf_backend/game_engine/executor.py#L16-L162)

### 3.3.1 阶段分发

`run_current_phase()` / `stream_current_phase()` 的分发关系：

- `night` -> 夜间技能结算
- `day_speech` -> 白天发言
- `voting` -> 白天投票
- `finished` -> 结束阶段

见：
- [werewolf_backend/game_engine/executor.py:40-63](werewolf_backend/game_engine/executor.py#L40-L63)

### 3.3.3 投票终局时不会再推进轮次

执行器与状态机当前都带有终局保护：

- 如果投票阶段内部已经判定游戏结束
- 则 `session["phase"] = finished`
- 且 `session["status"] = finished`
- 后续不会再执行 `voting -> night` 的阶段推进
- 因此不会出现“游戏已结束但 `round` 仍然额外 +1”的情况

这保证了：

- 对局轮次统计不会在投票终局时失真
- 复盘事件中的最终轮次与真实结束轮次一致
- 后续胜率、回合长度等统计口径不会被多加一轮污染


## 4. 胜负判定规则

当前胜负判定逻辑非常明确：

### 4.1 好人胜利

满足以下条件时，好人胜：

- 场上没有任何仍然“有效在场”的狼人

### 4.2 狼人胜利

满足以下条件时，狼人胜：

- 有效在场狼人数量 `>=` 有效在场非狼人数量

见：
- [werewolf_backend/game_engine/state_machine.py:53-62](werewolf_backend/game_engine/state_machine.py#L53-L62)

### 4.3 “有效在场”定义

当前不是简单看 `alive`，而是调用 `is_agent_active()`：

- 普通角色：必须 `alive`
- 白痴：`exiled` 也算仍在场

见：
- [werewolf_backend/game_engine/state_machine.py:48-52](werewolf_backend/game_engine/state_machine.py#L48-L52)

### 4.4 结束处理

游戏结束时：

- `phase` 设置为 `finished`
- `status` 设置为 `finished`
- `winner` 写入阵营名

见：
- [werewolf_backend/game_engine/state_machine.py:64-68](werewolf_backend/game_engine/state_machine.py#L64-L68)

## 6. Agent 决策输入的数据分层

当前系统不是“纯规则脚本自动执行全部技能”，而是采用：

- **规则层**：决定当前阶段能不能行动、有哪些合法动作、可选目标是谁
- **Agent / LLM 层**：在合法动作范围内，自主决定“选哪个动作 / 选谁 / 说什么”

也就是说，技能是否允许触发、触发时机、目标边界由后端规则控制；具体策略选择由 Agent 决策完成。

当前每次 agent 决策前，系统都会构造 `AgentDecisionInput`，其中信息被分成三层：

1. `public_state`：所有人都能看到
2. `private_state`：当前角色私有信息
3. `camp_shared_state`：阵营共享信息

构造见：
- [werewolf_backend/game_engine/decision_context.py:138-155](werewolf_backend/game_engine/decision_context.py#L138-L155)

## 6.1 public_state

包含：

- 当前阶段
- 当前回合
- 存活玩家列表
- `public_events`

见：
- [werewolf_backend/game_engine/decision_context.py:30-42](werewolf_backend/game_engine/decision_context.py#L30-L42)

## 6.2 private_state

包含：

- 自己的 id / role / camp / status
- 私有事实 `private_facts`
- 角色技能名（如果有）
- 女巫额外资源状态与夜晚上下文

见：
- [werewolf_backend/game_engine/decision_context.py:45-65](werewolf_backend/game_engine/decision_context.py#L45-L65)

## 6.4 决策归属：规则出发 vs Agent 自主决策

当前技能执行链路可以概括为两段：

### 第一段：规则先做边界约束

后端规则系统先构造 `legal_actions`，明确：

- 当前角色此刻是否允许行动
- 可用技能是什么
- 合法目标有哪些
- 是否存在额外约束
  - 女巫是否还有药
  - 当前狼刀目标是谁
  - 猎人是否因合法死因进入待开枪状态
  - 候选目标是否仍然存活

这一步属于“规则出发”，核心目的是防止 Agent 超规则行动。

### 第二段：Agent 在合法范围内做策略选择

在规则给出边界后，系统会把：

- `public_state`
- `private_state`
- `camp_shared_state`
- `legal_actions`

一起送入 `AgentDecisionInput`，再交给 Agent 的：

- `use_skill()`
- `vote()`
- `speak_streaming()`

由 Agent 自主决定：

- 是否救 / 毒 / 跳过
- 狼人刀谁
- 预言家查谁
- 猎人打谁
- 白天说什么
- 白天投谁

### 第三段：规则再做执行与结算

Agent 给出决策结果后，后端仍会继续负责：

- 校验决策是否落在合法范围内
- 把动作转成结构化事件
- 更新 `GameSession`
- 更新私有事实 / 公共事件
- 做死亡结算与胜负判定

因此当前系统的本质是：

- **规则负责边界与结算**
- **Agent 负责策略与选择**

而不是“规则直接替角色决定动作”。

## 7. 夜晚阶段链路

夜晚是当前最复杂的阶段。

入口见：
- [werewolf_backend/game_engine/actions/night.py:39-240](werewolf_backend/game_engine/actions/night.py#L39-L240)

## 7.1 夜晚开始时做什么

夜晚开始时会：

1. 强制设置 `session["phase"] = night`
2. 发送 `PHASE_CHANGE`
3. 初始化临时变量：
   - `wolf_target_id`
   - `wolf_votes`
   - `witch_heal_used`
   - `witch_poison_target_id`
4. 清掉上一夜残留的 `witch_potion_used`

见：
- [werewolf_backend/game_engine/actions/night.py:43-52](werewolf_backend/game_engine/actions/night.py#L43-L52)

## 7.2 夜晚行动白名单与顺序

当前夜晚只允许以下角色进入主动技能流程：

1. 狼人
2. 预言家
3. 女巫

并且顺序固定为：

- `狼人 -> 预言家 -> 女巫`

见：
- [werewolf_backend/game_engine/actions/night.py:24-25](werewolf_backend/game_engine/actions/night.py#L24-L25)
- [werewolf_backend/game_engine/actions/night.py:98-107](werewolf_backend/game_engine/actions/night.py#L98-L107)

这意味着：

- 猎人、村民、白痴不会在夜晚进入默认技能分支
- 夜晚不存在“未知角色兜底 use_skill()”的隐式逻辑

## 7.3 狼人夜晚技能链路

### 7.3.1 决策权限

狼人夜晚可执行 `kill`，并且目标列表会自动排除：

- 自己
- 狼队友

也就是说：

- “狼人这一晚能不能出手、能打谁”由规则系统决定
- “在合法目标里最终选择谁”由狼人 Agent 自主决策

见：
- [werewolf_backend/game_engine/decision_context.py:91-104](werewolf_backend/game_engine/decision_context.py#L91-L104)

### 7.3.2 协作机制

当前已经实现“全员提交 + 多数决”：

- 每个存活狼人各自进行一次技能决策
- 每次都会发出一条 `AGENT_SKILL`
- 后端收集所有非空 `target_id`
- 对狼刀目标计票
- 票数最高者成为最终狼刀目标
- 若最高票平票，则在平票目标中随机选一个

见：
- [werewolf_backend/game_engine/actions/night.py:28-36](werewolf_backend/game_engine/actions/night.py#L28-L36)
- [werewolf_backend/game_engine/actions/night.py:112-132](werewolf_backend/game_engine/actions/night.py#L112-L132)

### 7.3.3 与女巫的衔接

最后一只狼人行动后，系统会立即把聚合后的狼刀目标写入：

- `session["night_pending_target_id"]`

这样女巫在自己的单次决策窗口里，就能看到当夜狼刀目标。

见：
- [werewolf_backend/game_engine/actions/night.py:128-131](werewolf_backend/game_engine/actions/night.py#L128-L131)

## 7.4 预言家夜晚技能链路

### 7.4.1 决策权限

预言家夜晚可执行 `inspect`。

其目标为所有其他存活玩家。

也就是说：

- “预言家能不能查验、可以查谁”由规则系统给出
- “最终查验哪一名目标”由预言家 Agent 自主决策

## 7.5 女巫夜晚技能链路

### 7.5.1 决策模式

当前女巫已经改为“单次联合决策窗口”，不是先救再毒的串行双窗口。

也就是说，女巫每晚只会被问一次，在一次决策中选择：

- `heal`
- `poison`
- `skip`

这里同样分成两层：

- 规则先决定：今晚是否还能救、是否还能毒、狼刀目标是谁
- 女巫 Agent 再决定：在合法动作里最终选择 `heal / poison / skip` 的哪一个

对应实现见：
- [werewolf_backend/game_engine/actions/night.py:161-197](werewolf_backend/game_engine/actions/night.py#L161-L197)

### 7.5.2 女巫私有信息

女巫额外能看到：

- 解药是否可用 `heal_available`
- 毒药是否可用 `poison_available`
- 当前狼刀目标 `night_context.wolf_target_id`

见：
- [werewolf_backend/game_engine/decision_context.py:57-65](werewolf_backend/game_engine/decision_context.py#L57-L65)

### 7.5.3 女巫合法动作限制

女巫合法动作通过 `build_skill_legal_actions()` 控制。

关键规则如下：

#### 解药规则

女巫能救人的前提：

- 解药还没用过
- 当前存在狼刀目标
- 当夜还没用过其他药

当前实现允许女巫自救：

- 如果当夜狼刀目标正好是女巫自己
- 且解药仍可用
- 则女巫可以对自己使用解药

见：
- [werewolf_backend/game_engine/decision_context.py:105-118](werewolf_backend/game_engine/decision_context.py#L105-L118)

#### 毒药规则

女巫能毒人的前提：

- 毒药还没用过
- 当夜还没用过其他药

当前实现中毒药目标仍会排除自己，因此“允许自救”只适用于解药，不适用于自毒。

见：
- [werewolf_backend/game_engine/decision_context.py:91-104](werewolf_backend/game_engine/decision_context.py#L91-L104)
- [werewolf_backend/game_engine/decision_context.py:105-118](werewolf_backend/game_engine/decision_context.py#L105-L118)

#### 单夜单瓶药

当前严格限制：

- 同一夜不能同时使用解药和毒药
- 一旦使用任意一种药，就会设置 `witch_potion_used = True`
- 当夜后续不再允许另一种药

这是“单夜限制”。此外当前实现还带有“全局次数限制”：

- 解药全局只能使用 1 次
- 毒药全局只能使用 1 次
- 这两项通过女巫私有事实永久记录，而不是只靠当夜临时变量

见：
- [werewolf_backend/game_engine/decision_context.py:26-27](werewolf_backend/game_engine/decision_context.py#L26-L27)
- [werewolf_backend/game_engine/decision_context.py:57-60](werewolf_backend/game_engine/decision_context.py#L57-L60)
- [werewolf_backend/game_engine/decision_context.py:105-118](werewolf_backend/game_engine/decision_context.py#L105-L118)
- [werewolf_backend/game_engine/actions/night.py:167-188](werewolf_backend/game_engine/actions/night.py#L167-L188)

### 7.5.4 女巫结算方式

#### 选择解药时

- 标记 `witch_heal_used = True`
- 标记 `session["witch_potion_used"] = True`
- 写入女巫私有事实 `witch_heal_used`
- 发出 `AGENT_SKILL(skill="heal")`
- 实际治疗目标固定是当夜狼刀目标，而不是任意人
- 后续跨夜也不能再次使用解药

见：
- [werewolf_backend/game_engine/actions/night.py:167-178](werewolf_backend/game_engine/actions/night.py#L167-L178)

#### 选择毒药时

- 记录 `witch_poison_target_id`
- 标记 `session["witch_potion_used"] = True`
- 写入女巫私有事实 `witch_poison_used`
- 发出 `AGENT_SKILL(skill="poison")`
- 后续跨夜也不能再次使用毒药

见：
- [werewolf_backend/game_engine/actions/night.py:180-197](werewolf_backend/game_engine/actions/night.py#L180-L197)

## 7.6 猎人夜晚技能链路

猎人不属于夜晚主动行动白名单，不会常规进入夜晚技能循环。

当前猎人只能在“死亡触发”时被动触发开枪。

### 7.6.1 可触发开枪的死亡来源

当前只有两种情况允许猎人开枪：

- 夜间被狼刀死
- 白天被投票放逐死

### 7.6.2 明确禁止开枪的情况

以下情况不会触发猎人开枪：

- 被女巫毒死
- 没有进入待触发状态
- 已经开过枪
- 当前已没有合法存活目标

- [werewolf_backend/game_engine/actions/night.py:206-216](werewolf_backend/game_engine/actions/night.py#L206-L216)

## 7.7 夜晚最终死亡结算顺序

所有夜间角色行动完成后，系统才统一做死亡结算。

当前顺序是：

1. 根据狼人聚合结果得到最终狼刀目标
2. 若女巫没有救，则把狼刀目标加入死亡列表
3. 若女巫毒了别人，且该人不与狼刀目标重复，则加入死亡列表
4. 按顺序执行死亡
5. 若死者是因 `wolf_attack` 死亡的猎人，则立刻触发猎人开枪

见：
- [werewolf_backend/game_engine/actions/night.py:199-216](werewolf_backend/game_engine/actions/night.py#L199-L216)

### 7.7.1 对外公开的夜晚结果

夜晚对外公开的信息仍是：

- 每只狼自己的 `AGENT_SKILL`
- 预言家自己的 `AGENT_SKILL`
- 女巫自己的 `AGENT_SKILL`
- 最终 `AGENT_STATUS_CHANGE`

也就是说：

- 狼队内部是多 agent 协作投票
- 但外部结算仍然只消费一个最终狼刀目标

## 7.8 夜晚结束后的终局检查

夜晚死亡与技能结算结束后，系统立即计算胜负。

若已经分出胜负：

- 直接把 phase 设为 `finished`
- 发出 `PHASE_CHANGE(finished)`
- 发出 `GAME_OVER`

见：
- [werewolf_backend/game_engine/actions/night.py:221-229](werewolf_backend/game_engine/actions/night.py#L221-L229)

## 8. 白天发言阶段链路

## 8.1 执行顺序

白天发言阶段会：

1. 发出 `PHASE_CHANGE`
2. 按 `session["agents"]` 当前顺序遍历所有玩家
3. 仅允许 `status == alive` 的玩家发言
4. 每个玩家先流式输出 chunk，再输出最终完整发言

## 8.2 发言事件

每名玩家的发言会产生两类事件：

- `AGENT_SPEAK_CHUNK`：增量流式内容
- `AGENT_SPEAK`：最终完整内容

定义见：
- [werewolf_backend/game_engine/events.py:41-57](werewolf_backend/game_engine/events.py#L41-L57)

## 8.3 发言后的公共信息传播

完整发言事件会：

- 追加到 `public_events`
- 广播给所有 agent 的 `observe_public_event()`

这意味着后续投票阶段，所有存活 agent 都能利用白天公开发言做决策。

## 9. 投票阶段链路

入口见：
- [werewolf_backend/game_engine/actions/voting.py:16-163](werewolf_backend/game_engine/actions/voting.py#L16-L163)

## 9.1 参与投票的人

当前投票阶段只允许：

- `status == alive` 的玩家投票

见：
- [werewolf_backend/game_engine/actions/voting.py:91-106](werewolf_backend/game_engine/actions/voting.py#L91-L106)

### 9.1.1 当前白痴 exiled 的实际行为说明

虽然胜负判定把 `exiled` 白痴视为仍在场，但投票和白天发言阶段都只检查 `alive`。

因此当前实现下：

- 白痴被放逐后不会继续发言
- 白痴被放逐后不会继续投票
- 但在胜负计算里仍算有效在场角色

这属于当前实现中的一个明确特殊规则组合。

## 9.2 投票合法目标

投票候选人为：

- 所有其他存活玩家
- 不包含自己

见：
- [werewolf_backend/game_engine/decision_context.py:80-88](werewolf_backend/game_engine/decision_context.py#L80-L88)

## 9.3 投票事件

每个投票动作都会：

- 生成 `AGENT_VOTE`
- 追加到 `public_events`

见：
- [werewolf_backend/game_engine/actions/voting.py:101-106](werewolf_backend/game_engine/actions/voting.py#L101-L106)

## 9.4 平票规则

当前平票规则已经按标准需求改为“平安日”：

- 若最高票人数超过 1 人
- 则本轮无人出局
- 不进行二次投票
- 直接结束投票阶段

见：
- [werewolf_backend/game_engine/actions/voting.py:108-129](werewolf_backend/game_engine/actions/voting.py#L108-L129)

如果此时已经满足胜负条件，也会直接结束游戏。

## 9.5 非平票时的放逐规则

若最高票唯一：

- 该玩家成为出局目标

见：
- [werewolf_backend/game_engine/actions/voting.py:131-152](werewolf_backend/game_engine/actions/voting.py#L131-L152)

### 9.5.1 白痴被投票放逐

若被放逐的是白痴：

- 不进入 `dead`
- 改为 `exiled`
- 发出特殊 `AGENT_STATUS_CHANGE`
- 公开其白痴身份
- 同时标记：
  - `can_vote = False`
  - `can_speak = False`
  - `is_alive = True`

见：
- [werewolf_backend/game_engine/actions/voting.py:41-59](werewolf_backend/game_engine/actions/voting.py#L41-L59)
- [werewolf_backend/game_engine/actions/voting.py:133-136](werewolf_backend/game_engine/actions/voting.py#L133-L136)

### 9.5.2 普通角色被放逐

若不是白痴：

- 直接进入 `dead`
- 发出 `AGENT_STATUS_CHANGE(cause="vote_exile")`

见：
- [werewolf_backend/game_engine/actions/voting.py:30-39](werewolf_backend/game_engine/actions/voting.py#L30-L39)
- [werewolf_backend/game_engine/actions/voting.py:136-140](werewolf_backend/game_engine/actions/voting.py#L136-L140)

## 9.6 猎人白天开枪链路

若白天被放逐出局的是猎人：

1. 先把猎人本体状态写回全局 `GameSession`
2. 写入 `hunter_pending_shot_id`
3. 写入 `hunter_pending_shot_cause = vote_exile`
4. 重新计算猎人技能合法性
5. 仅当仍存在合法存活目标时，猎人才能执行一次 `shoot`
6. 发出 `AGENT_SKILL`
7. 被命中的目标会立刻同步写回 `GameSession` 为死亡状态

见：
- [werewolf_backend/game_engine/actions/voting.py:61-87](werewolf_backend/game_engine/actions/voting.py#L61-L87)
- [werewolf_backend/game_engine/actions/voting.py:131-152](werewolf_backend/game_engine/actions/voting.py#L131-L152)

## 9.7 投票结束后的终局检查

投票阶段结束后会再次检查胜负。

若已经分出胜负：

- phase 置为 `finished`
- 发出 `PHASE_CHANGE(finished)`
- 发出 `GAME_OVER`
- 不再推进到下一轮 `night`
- 不会执行额外的 `round + 1`

见：
- [werewolf_backend/game_engine/actions/voting.py:144-152](werewolf_backend/game_engine/actions/voting.py#L144-L152)

## 10. 角色技能链路总表

## 10.1 狼人

### 可见信息

- 公共信息
- 狼队队友列表（存活狼人）

见：
- [werewolf_backend/game_engine/decision_context.py:68-77](werewolf_backend/game_engine/decision_context.py#L68-L77)

### 主动技能

- 夜晚 `kill`

### 目标限制

- 不能选自己
- 不能选狼队友
- 只能选其他存活玩家

### 结算方式

- 每只狼单独提交
- 后端做多数决
- 平票随机
- 产生一个最终狼刀目标

## 10.2 预言家

### 可见信息

- 公共信息
- 自己历史查验私有事实

### 主动技能

- 夜晚 `inspect`

### 目标限制

- 可查验任意其他存活玩家

### 结算方式

- 公开发出一次 `AGENT_SKILL`
- 私下记录目标 `role` 和 `camp`

## 10.3 女巫

### 可见信息

- 公共信息
- 药剂资源状态
- 当前狼刀目标
- 自己历史用药记录

### 主动技能

- 夜晚单次联合决策：
  - `heal`
  - `poison`
  - `skip`

### 规则限制

- 单夜只能用一瓶药
- 解药只能救当夜狼刀目标
- 若女巫自己是当夜狼刀目标，则允许自救
- 毒药不能毒自己
- 解药全局只能使用 1 次
- 毒药全局只能使用 1 次
- 解药、毒药的全局使用次数通过女巫私有事实永久记录

### 结算方式

- `heal`：阻止狼刀目标死亡
- `poison`：额外造成一名目标死亡
- `skip`：不产生真实伤害变化

## 10.4 猎人

### 常规阶段

- 不属于夜晚主动技能白名单
- 白天正常发言、正常投票（前提是还活着）

### 被动触发技能

- `shoot`

### 触发条件

仅两类：

- 夜间死于 `wolf_attack`
- 白天死于 `vote_exile`

### 不触发情况

- 死于 `poison`
- 已经开过枪

### 决策归属

- “猎人此刻是否允许开枪、能打谁”由规则系统决定
- “在合法目标中最终开枪打谁”由猎人 Agent 自主决策

## 10.5 白痴

### 当前特殊逻辑

- 没有夜间主动技能
- 被投票放逐时不会立刻死亡
- 改为 `exiled`
- 会公开身份
- 不能继续发言、不能继续投票
- 但胜负判定仍视作有效在场

## 10.6 村民

### 当前能力

- 无夜间技能
- 白天发言
- 白天投票
- 完全基于公共信息决策

## 11. 当前完整回合流程示意

一个完整回合当前可概括为：

### 第 1 步：夜晚开始

- phase = `night`
- 发 `PHASE_CHANGE`

### 第 2 步：狼人行动

- 所有存活狼人各自刀人
- 发各自 `AGENT_SKILL`
- 聚合为最终狼刀目标

### 第 3 步：预言家行动

- 选择一名玩家查验
- 发 `AGENT_SKILL`
- 结果只写入预言家私有事实

### 第 4 步：女巫行动

- 在单次窗口中选择救 / 毒 / 不用药
- 发 `AGENT_SKILL`

### 第 5 步：夜晚死亡结算

- 结算狼刀
- 结算毒药
- 若猎人因狼刀死亡，触发开枪
- 发 `AGENT_STATUS_CHANGE`

### 第 6 步：夜晚结束胜负检查

- 若已结束，发 `PHASE_CHANGE(finished)` + `GAME_OVER`
- 否则进入白天发言

### 第 7 步：白天发言

- phase = `day_speech`
- 存活玩家依次发言
- 先发 chunk，再发最终发言
- 最终发言写入 `public_events`

### 第 8 步：白天投票

- phase = `voting`
- 存活玩家依次投票
- 发 `AGENT_VOTE`

### 第 9 步：投票结算

- 平票 -> 平安日，无人出局
- 唯一最高票 -> 放逐对应玩家
- 若被放逐者是猎人，触发开枪
- 若被放逐者是白痴，进入 `exiled`

### 第 10 步：投票后胜负检查

- 若结束，发 `PHASE_CHANGE(finished)` + `GAME_OVER`
- 若未结束，进入下一轮 `night`
- 只有在对局继续时才会 `round + 1`

## 12. 当前实现的几个关键特征

## 12.1 已经实现的规则一致性

当前后端已经明确实现并验证的关键点包括：

- 夜间角色只走白名单，不存在默认兜底技能分支
- 狼人是“全员提交 + 多数决 + 平票随机”
- 女巫是“单次联合决策窗口”
- 女巫“单夜单瓶药”
- 女巫解药全局只能使用 1 次
- 女巫毒药全局只能使用 1 次
- 女巫允许在自己成为狼刀目标时自救
- 女巫不能毒自己
- 投票平票为平安日，不二次投票
- 猎人仅在狼刀死 / 投票死时能开枪
- 猎人被毒死不能开枪
- 猎人开枪次数通过私有事实防重入
- 猎人开枪与女巫毒药都会提前过滤已死亡目标，避免生成无效技能事件

## 12.2 当前实现中的一个显式设计点

白痴当前属于“结算视为在场、行为上不可继续参与”的实现：

- 胜负计算视其仍有效
- 但发言与投票阶段不再参与

如果后续要进一步贴近某套更严格的桌游规则，需要单独确认这部分是否继续保持当前设计。

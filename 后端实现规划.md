# 后端实现规划

本文档用于指导 `werewolf_backend` 的下一步实现。这里的首要目标不是单纯做一个“能跑流程的游戏后端”，而是围绕课题要求，构建一个 **基于多 Agent 协作框架、能够完成信息不对称博弈的狼人杀 Agent Team 系统**。

根据 [课题要求.md](课题要求.md)，后端实现必须优先满足以下核心目标：

1. **每个 AI 玩家都是独立 Agent**
2. **每个 Agent 按角色拥有独立目标、策略与行动空间**
3. **在严格信息隔离约束下完成推理、发言与决策**
4. **后端具备完整对局引擎，负责回合流转与胜负裁决**
5. **全程输出结构化日志，保证对局过程可观测、可复盘、可评测**

因此，后端规划必须以“多 Agent 对抗系统”为中心，而不是以“WebSocket 演示链路”或“普通游戏服务端”作为核心出发点。

---

## 1. 重新明确后端任务目标

后端最终要实现的，不是“统一脚本替多个玩家输出文本”，而是：

- 在一局狼人杀中创建多个玩家实例
- 每个玩家实例绑定一个独立 Agent
- 每个 Agent 在整局生命周期内持续存在
- 每个 Agent 只接收其角色允许看到的信息
- 每个 Agent 独立完成：
  - 观察
  - 推理
  - 发言
  - 投票
  - 技能决策
  - 记忆更新
- 游戏引擎统一调度这些 Agent，并完成规则裁决

所以这里必须明确：

### 1.1 “每个 AI 玩家是一个独立 Agent” 是硬目标

这意味着后端不能只做：
- 按角色临时创建一个对象生成一句话
- 每轮临时生成输出然后丢弃上下文

而必须做到：
- `player_id -> agent instance`
- 每个 Agent 在整局中持续存在
- 每个 Agent 有自己的记忆、历史行为、可见状态和决策链路

### 1.2 游戏后端的核心是“调度多个独立 Agent 的对抗系统”

因此后端主线应是：

- 游戏引擎定义规则
- 会话层保存运行态
- Agent 层保存每个玩家的独立决策实体
- 模型层为 Agent 提供推理能力
- 日志层记录整个多 Agent 博弈过程

### 1.3 信息隔离不是附加项，而是系统约束

必须严格区分：

- **公共信息**：所有玩家都能看到
- **阵营共享信息**：如狼人夜晚协同信息
- **私有信息**：如预言家查验结果、女巫药剂状态
- **Agent 内部记忆**：仅该 Agent 自身可见

如果后端没有实现严格信息隔离，那就不符合课题目标。

---

## 2. 后端总体技术路线

本项目后端建议明确采用 **Python 技术栈**，因为课题重点在于：

- 多 Agent 决策系统实现
- 异步调度
- 规则引擎与信息隔离
- LLM / Agent 能力接入
- 结构化日志与评测扩展

推荐技术栈如下：

- **Python**：主开发语言
- **FastAPI**：HTTP 与 WebSocket 接入框架
- **asyncio**：异步调度基础
- **Uvicorn**：服务启动
- **Redis**：活跃对局运行态缓存
- **MySQL**：账号、统计、排行榜、评测结果持久化
- **SQLAlchemy**：MySQL 数据访问层
- **Pydantic Settings**：配置管理
- **LangChain**：模型调用编排层

### 2.1 Redis + MySQL 双存储架构

- **Redis**：保存游戏过程中的全部运行时信息
- **MySQL**：保存玩家账号、累计统计、排行榜、评测结果等长期业务数据

也就是说：

- **Redis 负责过程态**
- **MySQL 负责账号、统计与评测持久化**

不再建议把运行中对局主要落在关系型数据库中，也不建议把游戏过程完全留在 Python 进程内内存中。

## 3. 当前后端现状与差距

目前后端已经具备基础骨架，但距离课题目标还有明显差距。

### 3.1 当前已有的基础能力

- FastAPI 应用入口
- `GET /`
- `GET /api/health`
- `WS /ws`
- Redis session store 基础骨架
- 最小游戏状态模型与状态机骨架
- WebSocket 命令分发骨架
- `game_engine/actions/` 模块骨架
- `agent_team_core/` 角色 Agent 类骨架
- `llm_service/` mock / LangChain 可切换发言链路

### 3.2 当前最关键的未完成点

尽管已经有骨架，但目前还**没有真正达到课题要求中的“多 Agent 对抗系统”实现深度**。主要差距在于：

1. **还没有实现 player_id -> 长生命周期 Agent 实例管理**
2. **还没有为每个 Agent 建立独立 memory**
3. **还没有真正实现严格信息隔离输入构造**
4. **还没有把 vote() / use_skill() 接入 Agent 决策链**
5. **还没有完成真实夜晚技能结算与胜负判定**
6. **还没有完成结构化日志与复盘/评测数据落库**

所以当前实现只能算：

- 已完成“多 Agent 架构起步”
- 尚未完成“多 Agent 对抗系统落地”

## 4. 核心架构重心：从“流程后端”转向“多 Agent 对抗引擎”

后端的重点应重新排序，优先保证以下四个系统成立：

## 4.1 多 Agent 生命周期系统

必须建立：

- `game_id -> player_id -> agent instance`

每局开始时：
- 创建玩家列表
- 为每个玩家分配身份
- 为每个玩家创建一个独立 Agent 实例
- 让 Agent 在整局中持续存在

这一步是整个课题后端的第一性能力。

## 4.2 信息隔离输入系统

每次调用 Agent 前，后端必须先构造：

- `public_state`
- `private_state`
- `camp_shared_state`
- `agent_memory`

例如：
- 狼人 Agent 能看到狼队友
- 预言家 Agent 能看到自己的查验结果
- 村民 Agent 只能看到公共发言与公共死亡信息

不能让所有 Agent 接收到几乎相同的输入，否则就失去课题的核心研究价值。

## 4.3 游戏规则调度系统

后端游戏引擎必须负责：

- 角色分配
- 夜晚阶段推进
- 白天发言阶段推进
- 投票阶段推进
- 技能结算
- 死亡/放逐状态更新
- 胜负判定

Agent 只负责“做决策”，不负责“改状态”。

## 4.4 结构化可观测系统

后端必须在早期就输出结构化日志，而不是后补。

至少要记录：
- 对局元信息
- 轮次变化
- 阶段变化
- Agent 发言
- Agent 投票
- Agent 技能行为
- 状态变化
- 最终胜负结果

这是课题要求中的硬条件，也是后续复盘和评测的基础。

## 5. 分层设计建议

为了避免职责混乱，建议把后端分为六层：

### 5.1 接入层
- FastAPI
- WebSocket
- Uvicorn

职责：
- 提供 HTTP 入口
- 提供 WebSocket 连接入口
- 接收前端控制命令
- 向前端推送结构化事件

### 5.2 会话与运行态层
- Redis
- `cache/session_store.py`
- `cache/serializers.py`

职责：
- 保存活跃对局运行态
- 保存当前 phase / round / status
- 保存临时动作、暂停/停止标志、等待输入状态

### 5.3 游戏引擎层
- `game_engine/models.py`
- `game_engine/state_machine.py`
- `game_engine/engine.py`
- `game_engine/actions/`
- `game_engine/win_checker.py`

职责：
- 定义规则真值状态
- 调度阶段流转
- 调用 Agent
- 执行规则结算
- 维护胜负逻辑

### 5.4 Agent 层
- `agent_team_core/base_agent.py`
- `agent_team_core/factory.py`
- 各角色 Agent 实现

职责：
- 表示每个玩家的独立智能体
- 管理玩家自己的记忆
- 接收信息隔离后的输入
- 产生发言、投票、技能决策

### 5.5 模型编排层
- LangChain
- `llm_service/client.py`
- `llm_service/prompt_builder.py`
- `llm_service/response_parser.py`
- `llm_service/chains/`

职责：
- 提供统一模型调用入口
- 管理发言/投票/技能的 Prompt Template
- 解析结构化输出
- 支持后续更换模型供应商

这里必须明确：
- **LangChain 不是游戏主控层**
- **LangChain 只是 Agent 的模型调用工具层**

### 5.6 持久化与评测层
- MySQL
- `database/models.py`
- `database/repositories.py`

职责：
- 保存账号
- 保存累计统计
- 保存排行榜数据
- 保存评测结果
- 为后续“评测+复盘”方向预留数据基础

## 6. 重新定义实现优先级

当前最重要的不是继续补更多演示动作，而是把课题主线拉直。

### P0：必须尽快完成
1. **玩家级独立 Agent 生命周期管理**
2. **信息隔离输入构造器**
3. **Agent memory 结构设计**
4. **游戏状态机与规则真值模型稳定化**

### P1：形成最小多 Agent 对抗闭环
1. `speak()` 走 Agent 决策链
2. `vote()` 走 Agent 决策链
3. 狼人夜晚决策链打通
4. 最小胜负判定打通

### P2：规则能力完善
1. 预言家查验
2. 女巫解药/毒药
3. 猎人开枪
4. 完整投票放逐与死亡结算

### P3：结构化日志与持久化
1. 对局结构化日志写入
2. MySQL 账号与统计模型
3. 对局结束后的统计写回
4. 为复盘/评测预留表结构

### P4：模型与策略增强
1. LangChain 真正用于 speak / vote / skill 全链路
2. 不同角色独立 Prompt
3. 策略与输出稳定化

### P5：课题进阶方向支持
1. 评测体系
2. 复盘系统
3. Leaderboard
4. 自进化 Agent 循环

首个版本不应再仅仅定义为“WebSocket 演示打通”，而应定义为：

## 9. 当前实现与课题目标的关系判断

当前已经实现的部分，适合作为后端起点，但必须明确：

- 当前实现 **符合方向**
- 当前实现 **不等于已经实现课题目标**

更准确地说：

### 已经完成的
- WebSocket 命令链路骨架
- Redis 会话层骨架
- 游戏状态模型骨架
- Agent 层骨架
- LangChain 接入骨架

### 还没有完成的关键目标
- 每个玩家持久存在的独立 Agent 实例
- 每个 Agent 的独立 memory
- 严格信息隔离输入
- vote / skill 决策链
- 完整技能与胜负逻辑
- 结构化日志与评测落库

因此，后续实现必须围绕这些缺口继续推进，而不是继续在演示链路上做局部细化。

## 10. 结论

后端接下来的核心任务，不应理解成：

- “继续补几个事件”
- “继续把 WebSocket 做复杂一些”

而应理解成：

> 构建一个在严格信息隔离下，由多个独立 Agent 参与推理、发言、投票、技能决策，并由游戏引擎统一裁决的狼人杀多智能体对抗系统。

因此，后续实现顺序应调整为：

1. 先补玩家级独立 Agent 生命周期
2. 再补信息隔离输入与 Agent memory
3. 再补 vote / skill 决策链
4. 再补完整规则结算与胜负裁决
5. 再补日志、持久化、评测与复盘

只有按这个顺序推进，后端实现才真正与课题目标对齐。
